## **עבודה עם Claude Code**
### מיפוי יכולות
כדי לעבוד טוב עם Claude Code, צריך קודם להבין מה הוא באמת יודע לעשות.
לא לפי שיווק, ולא לפי דוגמאות יפות, אלא לפי התנהגות בפועל.
Claude Code חזק במיוחד בהבנה של בסיסי קוד קיימים.
הוא יודע לקרוא הרבה קבצים, לזהות מבנים, ולעקוב אחרי זרימות לוגיות לאורך המערכת.
זה היתרון המרכזי שלו.
הוא טוב גם בעבודה עם טרמינל.
הרצת פקודות, חיפוש, בדיקה של מצבים קיימים.
הוא לא רק כותב קוד, הוא חוקר.
לעומת זאת, הוא פחות חזק בהחלטות מוצריות.
הוא לא יודע מה חשוב באמת לעסק.
הוא לא מבין אילוצים ארגוניים שלא כתובים בקוד.
ואם לא אומרים לו אחרת, הוא מניח שהכול פתוח לשינוי.
מיפוי יכולות נכון אומר לדעת:
•	איפה כדאי לשחרר אותו
•	איפה צריך להחזיק קצר
•	ואיפה הוא צריך לעבוד רק כהצעה, לא כמבצע
מתכנת שמכוון עבודה לא שואל “מה Claude Code יכול לעשות”,
אלא “מה נכון שהוא יעשה בפרויקט הזה”.
וברגע שההבחנה הזו ברורה,
הרבה תסכולים פשוט לא מופיעים.
 
### תצורות מומלצות
ההבדל בין עבודה מתסכלת עם Claude Code לבין עבודה יציבה וטובה
לא נמצא ביכולות שלו, אלא בתצורה שבה משחררים אותו לעבוד.
ברירת המחדל כמעט תמיד רחבה מדי.
יותר מדי קבצים.
יותר מדי חופש.
ופחות מדי גבולות.
תצורה טובה מתחילה בצמצום.
להגדיר מראש:
•	איזה תיקיות רלוונטיות למשימה
•	אילו קבצים לקריאה בלבד
•	ואיפה כל שינוי מחייב עצירה ואישור
בפרויקטים אמיתיים, עדיף להתחיל מתצורה שמרנית.
לתת לו לקרוא הרבה,
אבל לבצע מעט.
ככל שהעבודה מתקדמת,
אפשר להרחיב בהדרגה.
לא בבת אחת.
עוד מרכיב חשוב הוא קצב.
לא לתת לו לרוץ על כמה דברים במקביל.
משימה אחת. כיוון אחד. רצף ברור.
מתכנת שמכוון עבודה לא מחפש את התצורה “הכי חופשית”.
הוא מחפש את זו שנותנת את היחס הכי טוב
בין מהירות לשליטה.
וברגע שמוצאים אותה,
Claude Code הופך מכלי חזק אבל מסוכן
לשותף עבודה יציב שאפשר לסמוך עליו.


### דוגמאות עבודה מהשטח
הערך האמיתי של Claude Code מתגלה לא בדוגמאות מושלמות,
אלא בעבודה יומיומית, עם קוד שכבר קיים.
למשל, תיקון באג בפרויקט ותיק.
במקום לבקש “תמצא את הבעיה”,
העבודה מתחילה בהכוונה:
איזה אזור חשוד,
איפה כבר בדקנו,
ומה נחשב שינוי סביר.
Claude Code סורק, מריץ חיפושים, בודק הנחות.
אבל בכל שלב יש עצירה.
בדיקת כיוון.
ווידוא שהוא עדיין פותר את הבעיה הנכונה.
בדוגמה אחרת, הוספת פיצ’ר קטן למערכת קיימת.
לא נותנים לו לבנות הכול לבד.
נותנים לו להכין הצעה.
לבחון איך זה משתלב בקוד הקיים.
ואז מחליטים אם ואיך להמשיך.
בכל הדוגמאות האלה חוזר אותו דפוס:
הסוכן עובד,
אבל אתה מנהל.
לא נותנים לו “להוכיח את עצמו”.
לא בודקים כמה מהר הוא יכול לרוץ.
בודקים כמה מדויק הוא כשיש לו מסגרת.
וכשזה עובד נכון,
Claude Code מפסיק להיות כלי שמנסים בזהירות,
והופך לחלק טבעי מהעבודה.
